# AGENTS.md

### 0. 目标
本文件定义本仓库的默认协作规则，目标是：  
**最小上下文、可验证交付、低风险改动、可回滚。**

---

### 1. 优先级与冲突处理（MUST）
1. 用户当前指令 > 本文件规则。  
2. 若规则冲突，遵循“更具体、范围更小”的规则。  
3. 任何高风险或高影响改动，先澄清再实现。  

---

### 2. 工作流程（MUST）
1. **先给方案，再写代码**：先说明实现方法、影响范围、验证方式。  
2. **有歧义先澄清**：需求不清、风险高、影响大时，先提问题并等待确认。  
3. **Plan 只写方案**：不在 Plan 中写实现细节代码。  
4. **坚持 Spec Coding**：基于明确需求、约束、验收标准实现；不做随意试探式编码。  
5. **完成后给最短结论**：输出“结论 + 最直接可执行建议 + 下一步”。  

---

### 3. 任务拆分与协作（SHOULD）
> 说明：若运行环境支持多 Agent，按本节执行；否则由单 Agent 按同样分工顺序完成。

1. 先指定一个 Agent 产出总体 plan。  
2. 基于 plan 拆分低耦合子任务，定义每个子任务的输入、输出、验收标准。  
3. 子任务可并行则并行，不可并行则串行。  
4. 每个子任务只提供“最小必要上下文”，避免一次性注入过量背景。  
5. 任务完成后指定一个 Agent 统一汇总：目标、结果、验证、风险、后续建议。  

---

### 4. 编码规则（MUST）
1. 代码与标识符仅使用英文。  
2. 注释不写过程性废话（如“这里我改了什么”）；只写必要原因与约束。  
3. 规格描述优先使用概念定位（模块/职责/接口），不依赖行号。  
4. 非必要不改无关代码，不做“顺手重构”。  

---

### 5. 范围控制（MUST）
1. 默认仅关注以下目录：`xtuner/v1/`、`tests/`、`examples/`。  
2. 其他目录默认不读不改；如必须改动，需先说明必要性与影响。  
3. RL 相关改动优先在 `xtuner/v1/rl/` 内闭环完成。  

---

### 6. RL 目录导航（SHOULD）
- `xtuner/v1/rl/trainer/`: train worker、train controller、pack、old logprobs 相关逻辑  
- `xtuner/v1/rl/rollout/`: rollout backend 与 worker/controller  
- `xtuner/v1/rl/loss/`: GRPO / OREAL 等 loss 逻辑  
- `xtuner/v1/rl/agent_loop/`: agent loop 与任务流  
- `xtuner/v1/rl/judger/`: reward/correctness 评估  
- `xtuner/v1/rl/utils/`: RL 公共工具与 Ray 辅助  
- `xtuner/v1/rl/replay_buffer.py`: 中心化数据管理  
- `xtuner/v1/rl/evaluator.py`: 评估方法定义  

---

### 7. 质量与验证（MUST）
1. 最低质量标准：**可运行、可验证、可回滚**。  
2. 优先验证关键路径和高风险改动。  
3. Bug 流程必须是：**先复现 -> 再修复 -> 再验证**。  
4. 每个子任务需给出独立验证结果（通过/失败/风险）。  

---

### 8. 知识沉淀（SHOULD）
1. 同类流程重复出现 ≥3 次，沉淀为可复用 Skill。  
2. 被纠正后要记录原因并更新规则，避免重复错误。  

---

### 9. 最终汇报模板（MUST）
最终报告至少包含：
1. 任务目标  
2. 子任务划分与结果  
3. 验证结论（含命令/方法）  
4. 遗留风险  
5. 后续建议  
6. 一句话最短结论（可直接执行）
